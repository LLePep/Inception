Docker:

-Qu'est ce que un deamon?

Un daemon est un prog qui s'execute en arriere plan, il est lance souvent au demarrage avec les droits de roots si il appercoit une commande ou une requete specifique 

il execute ce que pourquoi il est programme.

exemple: cron, http, syslog

-Pourquoi Docker se differencie des VMs?

Docker se differencie des Vms car il n'emule pas de materiel de la machine. Il agit comme un daemon qui utilise la machine facon processus.

Sa conception repose sur namespace, Cgroups, PID1.

Les namespace servent a encapsuler notre Docker et a donner acces que a ce qu'il a sur lui (son namespace)

Cgroups servent a definir combien de ressource son alloue a un processus, des que le process s'enballe on peut avoir un l'OOM Killer (Out Of Memory)

PID1 l'isolation provoque par les namespace servent aussi a faire croire que notre daemon

-Qu'est ce que le PID1 ?

Le PID1 est le process parent de tous les process quand on boot notre os.

-Difference entre Alpine et Debian pour Docker:
Alpine est une distribution Linux qui utilise musl fortement utilise pour la securite et la legerete, alors que debian elle utilise glibc
dans notre cas on se souciera pas des effets de securite que apporte alpine et on utilisera tout dememe une version distroless de Debian pour avoir tout de meme la conscience que les failles de securite sont problematiques.

-Musl vs glibc les difference:
glibc est plus lourde a compiler que musl mise a jour depuis une decennie presque, si elle est plus lourde elle est bien plus adaptable a bcq de systeme et optimise pour la compilation de projet moderne,
tandis que musl est tout l'inverse plus legere, plus rapide, mais bien moins adaptable, elle est presente sur de projet plus leger et surtout niveau securite elle devient optimal.

NOTION:

le "shell and exec form", le shell form execute de facon brut du code de la
meme maniere que l'on pourrai le faire dans un terminal, alors que le exec form,
il utilise un json(where each element in the array is a command, flag, or argument) ou il fera appel a une commande shell specifique
deplus la gestion des signaux est bien meilleur, car mon application est directement le PID 1, hors avec un shell exec on tombe sur le PID 2
si docker envoi un SIGTERM /bin/sh ne le propage pas et l'application ne libere pas les ressources c'est donc un SIGKILL qui est envoye apres
INSTRUCTION ["executable","param1","param2"] (exec form)
INSTRUCTION command param1 param2            (shell form)


FROM

Sert a initialiser une nouvelle session de construction de docker image, cette image servira de base pour la suite des instructions
il suffit de specifier une distribution pour notre image.

RUN

RUN creer une couche sur notre image qui servira quand notre "docker build" s'effectue pour l'execution des commandes
mises dedans on prefere execute qu'un seul appel a notre manager de packes pour avoir moins de layer dans nos images 

ENTRYPOINT && CMD

Ces deux mots cles sont a differencier mais ils travaillent ensemble. ENTRYPOINT sert surtout defnir la commande
principal (on surcharge rarement option: --entrypoint), CMD est tout l'inverse elle est surtout la, pour
surcharge les arguments de la commande principal tres facilement.

LABEL

Sert a creer des metadonnees dans notre image docker, pour acceder a ces donnees il suffit
de taper "docker inspect <image>"

ENV

Sert a set les variables d'environnements a l'interieur de mon container, elles peuvent etre surcharge
avec l'option -e

ARG

Sert a set des variables de build-time, elle est donc temporaire lors de la creation mon container:
attention la Doc officiel conseille de ne pas mettre des donnees sensible lors de la creation car
celle-ci peuvent etre de nouveaux recuperer grace a un docker history <image>
Ps: different des variables d'environnements !

ADD && COPY

Elles sont tres similaires mais ADD gere les URL, les extractions aussi, les deux permettent de copier un espace
de notre architecture dans le container.
PS: Regle utiliser COPY car ADD est plus pousse

VOLUME

Sert a la creation de volume dans notre container si aucun point d'accroche est specifier 
docker gere le volume a sa facon.
a eclaicir: docker run -v ou un docker-compose.yml

WORKDIR

Sert a se positionner dans un repertoire lors de l'exec du container et pendant le build.

EXPOSE

Expose est avant tout une indication, il ne faut pas l'associer directement au mappage des ports
de notre machine. le mappage ce fait d'abord avec l'option -p, neanmoins il a tout de meme un effets
tres sympa sur un network de docker si le port sont bien mapper avec notre EXPOSE la communication peut alors
s'effectuer
