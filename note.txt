-Pourquoi Docker se differencie des VMs?

Docker se differencie des Vms car il n'emule pas de materiel de la machine. Il agit comme un daemon qui utilise la machine facon processus.

Sa conception repose sur namespace, Cgroups, PID1.

Les namespace servent a encapsuler notre Docker et a donner acces que a ce qu'il a sur lui (son namespace)

Cgroups servent a definir combien de ressource son alloue a un processus, des que le process s'enballe on peut avoir un l'OOM Killer (Out Of Memory)

PID1 l'isolation provoque par les namespace servent aussi a faire croire que notre daemon est le PID1

-Qu'est ce que le PID1 ?

Le PID1 est le process parent de tous les process quand on boot notre os.

-Difference entre Alpine et Debian pour Docker:
Alpine est une distribution Linux qui utilise musl fortement utilise pour la securite et la legerete, alors que debian elle utilise glibc

glibc est plus lourde a compiler que musl mise a jour depuis une decennie presque, si elle est plus lourde elle est bien plus adaptable a bcq de systeme et optimise pour la compilation de projet moderne,
tandis que musl est tout l'inverse plus legere, plus rapide, mais bien moins adaptable, elle est presente sur des projets plus leger et surtout niveau securite elle devient optimal.

Le bridge dans l'aspect reseau est le fait de connecter une carte reseau virtuelle ou physique, a une autre carte reseau pour faire en sorte qu'il partage le flux entre eux

Docker Network vs Host Network

Docker Network (Mode Bridge Docker) :
Le Docker Network fonctionne grâce à un mécanisme de NAT.
Il crée un sous-réseau virtuel isolé où chaque conteneur possède sa propre adresse privée.
Pour l'intercommunication, nos conteneurs communiquent entre eux en utilisant directement ces adresses privées ou le DNS interne créé par Docker (ex: mariadb).
Ce fonctionnement repose directement sur l'utilisation des Network Namespaces de Linux, qui permettent d'isoler la pile réseau de chaque conteneur.
L'avantage principal est que nos conteneurs sont totalement dissociés de la carte réseau de la machine hôte :
il est donc possible d'ouvrir les mêmes ports (ex: le port 80) dans plusieurs conteneurs différents sans aucun conflit.

Host Network:
Dans cette deuxieme configuration qui serai en l'occurence la configuration "vanilla".
Nos containers font directement parties des process de notre VM, ou docker est le parent,
Le desavantage de sa c'est que nos container vont directement prendre donc les port physiques de notre machine hote.

# Broken reverse DNS slows down connections considerably and name resolve is
# safe to skip if there are no "host by domain name" access grants

#skip-name-resolve
Le fichier de configuration de MariaDB indique qu'un Reverse DNS manquant ou défaillant ralentit considérablement les connexions
entre la base de données et les services qui l'interrogent. Dans notre cas,
le Reverse DNS n'est pas à proprement parler "cassé" :
c'est simplement que Docker fait le choix de ne pas gérer la résolution inversée (IP ➡️ Nom).
Pourtant, Docker possède bien un serveur DNS interne capable de faire correspondre les noms de services à leurs adresses IP (résolution directe).
Le problème technique sous-jacent qui empêche Docker de faire l'inverse est l'ambiguïté de la réponse :
sur quel nom le DNS devrait-il se baser pour identifier le conteneur ? L'ID unique du conteneur ?
Le nom du service dans le docker-compose.yml ? Ou le nom réel du conteneur ?
De plus, notre infrastructure Docker Network étant un réseau privé, étanche et entièrement contrôlé,
l'attribution des IP est verrouillée par le moteur Docker lui-même. Le risque d'usurpation d'identité réseau est donc nul.
Activer la directive skip-name-resolve permet donc de sauter cette vérification inutile et d'éliminer les temps d'attente de connexion.

bind-address            = 127.0.0.1
Configuration de l'écoute réseau (bind-address) :
La directive bind-address définit sur quelle adresse IP le service MariaDB doit ouvrir ses écoutes.
Par défaut, elle est configurée sur 127.0.0.1 (localhost),
ce qui isole MariaDB à l'intérieur de son propre conteneur et empêcherait WordPress de le contacter.
Dans un environnement Docker, l'adressage IP des conteneurs est dynamique et géré à chaque démarrage par le Docker Network.
Il est donc impossible de prédire et d'écrire une IP fixe dans notre fichier de configuration.
En remplaçant cette valeur par 0.0.0.0, on indique à MariaDB d'écouter sur toutes les interfaces réseau disponibles du conteneur.
Ainsi, peu importe l'IP dynamique que Docker attribue au conteneur,
MariaDB sera capable de recevoir et de traiter les requêtes provenant de WordPress

GESTION DES RESSOURCES:

1. key_buffer_size = 128M (La RAM pour les index)
    Qu'est-ce que c'est ? C'est la quantité de mémoire RAM allouée pour stocker les index des tables de type MyISAM (un vieux moteur de stockage de MySQL/MariaDB).
    Le rôle : Les index permettent à MariaDB de trouver une information instantanément sans relire toute la base de données. Plus ce buffer est grand, plus MariaDB garde d'index en RAM, et plus les recherches sont rapides.
    Pour ton projet : WordPress utilise aujourd'hui un moteur plus moderne appelé InnoDB. Tu peux donc laisser cette ligne commentée, la valeur par défaut suffit largement.

2. max_allowed_packet = 1G (La taille maximale d'une ligne)
    Qu'est-ce que c'est ? C'est la taille maximale d'une seule requête SQL ou d'une seule ligne de données que le serveur accepte de recevoir ou d'envoyer d'un coup.
    Le rôle : Si tu as un article WordPress extrêmement long, ou si tu tentes d'importer une sauvegarde complète de ton site qui fait plusieurs mégaoctets en une seule commande SQL, MariaDB a besoin d'un "paquet" assez grand pour la traiter.
    Pour ton projet : 1G (1 Gigaoctet) est une valeur énorme et très confortable. Cela évite les erreurs du type "MySQL server has gone away" lors des gros imports de bases de données.

3. thread_stack = 192K (La mémoire par question)
    Qu'est-ce que c'est ? C'est la taille de la mémoire de travail (la pile) allouée à chaque "thread" (chaque tâche).
    Le rôle : À chaque fois que WordPress pose une question à MariaDB, le serveur crée un petit calcul en interne. La thread_stack est l'espace mémoire réservé pour exécuter ce calcul ou ces fonctions SQL complexes.
    Pour ton projet : 192K est la valeur standard de sécurité sous Debian pour que le serveur ne manque pas de mémoire sur des requêtes classiques.

4. thread_cache_size = 8 (L'hôtel des employés)
    Qu'est-ce que c'est ? Au lieu de détruire et de recréer un "employé virtuel" (un thread) à chaque fois qu'un utilisateur clique sur ton site, MariaDB va en garder quelques-uns en veille dans un "cache".
    Le rôle : Créer un thread sous Linux prend du temps et de l'énergie au processeur. Avec un cache de 8, lorsqu'un utilisateur quitte ton site, son "employé" ne meurt pas : il attend sur une chaise que le prochain visiteur arrive pour le servir instantanément.

5. myisam_recover_options = BACKUP (La trousse de secours)
    Le rôle : Si ta VM plante brutalement (coupure de courant, crash de Docker) pendant que MariaDB écrivait dans une vieille table MyISAM, les fichiers peuvent être corrompus. Au redémarrage suivant, cette option dit à MariaDB : "Essaie de réparer automatiquement la table cassée, mais fais d'abord une copie de sauvegarde (BACKUP) des fichiers abîmés au cas où la réparation échouerait".

6. max_connections = 100 (La jauge à l'entrée 🚧)
    Qu'est-ce que c'est ? C'est le nombre maximum de clients simultanés (comme des processus WordPress) autorisés à se connecter en même temps à la base de données.
    Le rôle : Si 101 personnes tentent de charger une page de ton site en même temps, le 101ème recevra une erreur "Too many connections". Cela sert de fusible pour éviter que MariaDB ne consomme toute la RAM de ta VM et ne fasse crasher tout le système en voulant répondre à trop de monde à la fois.

7. table_cache = 64 (ou table_open_cache) (Les tiroirs ouverts)
    Qu'est-ce que c'est ? C'est le nombre maximum de tables de ta base de données que MariaDB peut garder "ouvertes" en même temps dans sa mémoire.
    Le rôle : Ouvrir un fichier sur le disque dur de la VM est une action lente. Si MariaDB garde les fichiers des tables WordPress ouverts dans son cache (ici 64 tables max), il peut y lire et y écrire beaucoup plus rapidement.

Pourquoi c'est une option avancee, dans le fonctionnement de base de linux quand un process change de proprio comme c'est le cas
avec le process php-fpm qui demarre en root et qui passe sur un autre user name "wwww-data". Normalement la dump de memoire n'est pas
autorise elle est enfaite verrouillé donc l'option mise a "yes" enleve cette securite. C'est une option qui peut etre utile sur une installation en dev mais 
absolument a eviter sur de la prod.

Apres ca on a un petit commentaire qui explique les 3 different modes (static, dynamic, ondemand).
Le static comme son nom l indique est la pour avoir un server rapide qui consommera tjs la meme quantité de ressource, il est bien dans le cas de forte demande continue ou si l'on veux bcq de performance
en contrepartie de l'economie de RAM faite sur les heure creuses
dynamic: il creer/detruit des child selon la demande qui vient c'est une bonne option pour les infras de taille moyenne et qui fluctuent.
Ondemand: donc "Sur demande" c'est a dire qui creer/detruit des childs que sur demande/absence c'est un bon mode pour les petits infras.
